# 研报撰写标准作业程序（SOP）

> 版本：v1.0  
> 生效日期：2026-04-02  
> 适用范围：投研助手Workspace内所有深度研报产出

---

## 一、流程总览

```mermaid
graph LR
    A[1.选题立项] --> B[2.数据获取]
```

## 二、分步操作规范

### 步骤1：选题立项（Topic Selection）

| 要素 | 操作说明 |
|------|----------|
| **选题来源** | ① 宏观政策突发变化；② 行业景气度拐点信号；③ 产业链调研异常值；④ 市场热点与认知差。 |
| **立项评估** | 填写 `.agentstalk/` 下的选题立项单，明确：研究对象、时间维度、核心假设、预期产出、所需数据源。 |
| **可行性判断** | 评估数据可获取性、分析周期、结论可操作性。不可行则即时退回或缩小范围。 |

**交付物**：选题立项单（Markdown格式，存放于 `.agentstalk/`）

---

### 步骤2：数据获取（Data Acquisition）

| 数据类型 | 获取方式 | 质量要求 |
|----------|----------|----------|
| **宏观政经数据** | Wind、Bloomberg、国家统计局、央行官网、重要会议通稿 | 核对发布机构与发布时间，优先使用官方原始数据 |
| **行业运行数据** | 行业协会月报、百川盈孚、SMM、海关进出口数据 | 注明数据口径（产量/销量/装机量），区分预测值与官方值 |
| **公司财务数据** | 交易所公告、Wind、Choice、公司年报/季报 | 使用审计后数据，业绩预告需标注为" preliminary " |
| **调研与高频数据** | 产业链访谈、卫星/高频指标（如物流、开工率） | 标注样本量、调研时间、数据推算假设 |

**操作规范**：
1. 所有原始数据必须留存本地副本（CSV/JSON），路径统一为 `data/raw/YYYY-MM-DD_数据描述.csv`。
2. 数据清洗脚本需同步提交至 `scripts/data-pipeline/`。
3. 若数据存在缺失或口径变更，需在 `.agentstalk/` 中向分析Agent发出数据勘误通知。

**交付物**：原始数据文件 + 清洗脚本 + 数据说明文档

---

### 步骤3：基本面分析（Fundamental Analysis）

#### 3.1 宏观政经分析
- 定位当前所处的政策周期（宽松/收紧/结构性）
- 识别对目标行业有直接传导效力的宏观变量（利率、汇率、信贷、财政支出方向）
- 评估地缘政治与监管变动的风险敞口

#### 3.2 行业基本面分析
- **供需模型**：绘制产业链图谱，量化产能利用率、库存周期、价格弹性
- **竞争格局**：CR5、市场份额变化、技术路线替代风险
- **利润分布**：上中下游利润分配、议价能力变化

#### 3.3 公司基本面分析（如适用）
- 核心财务指标：ROE、毛利率、净利率、经营现金流/净利润
- 相对估值：PE、PB、EV/EBITDA 与历史分位、同业比较
- 成长质量：收入增速 vs 利润增速、资本开支强度

**操作规范**：
- 所有核心结论必须基于至少两种独立数据源交叉印证。
- 使用占位数据时，必须在段落末尾或图表下方以 **"【数据假设】"** 格式明确标注。

**交付物**：分析草稿（Markdown）+ 关键数据表/图

---

### 步骤4：交叉验证（Cross-Validation）

| 验证维度 | 验证内容 |
|----------|----------|
| **数据交叉验证** | 同一指标是否在不同数据源间保持一致；若存在偏差，解释原因并选择可信源 |
| **逻辑交叉验证** | 宏观推断 → 行业演绎 → 公司结论 是否形成自洽链条；是否存在反例 |
| **历史回测验证** | 将当前核心假设代入历史类似情景，检验推演逻辑的稳健性 |
| **外部Agent复核** | 委托独立的宏观分析Agent或数据抓取Agent对关键结论进行复核 |

**操作规范**：
- 若交叉验证发现重大矛盾，必须返回步骤2或步骤3修正，并在 `.agentstalk/` 中记录修正原因。
- 验证结果形成《交叉验证备忘录》，附于研报末尾或存入 `SOP/validation-notes/`。

**交付物**：交叉验证备忘录

---

### 步骤5：研报撰写（Report Writing）

#### 5.1 内容框架
严格按照 `templates/report-template.html` 的章节结构撰写：

1. **封面**：标题、作者、日期、免责声明
2. **投资要点摘要**：3-5条核心结论，突出认知差与操作建议
3. **宏观政经分析**：政策周期、流动性环境、宏观→行业映射
4. **行业/公司基本面分析**：产业链、供需格局、财务指标
5. **数据图表区**：时间序列图、比较图、核心数据表
6. **风险提示**：政策、市场、数据、模型四类风险
7. **结论与操作建议**：明确评级、目标价/区间、风控建议

#### 5.2 写作规范
- **术语**：统一使用金融与产业标准术语（如"升贴水""产能利用率""风险敞口""信贷周期"）
- **数据引用**：所有数据标注来源及截止日期
- **逻辑密度**：拒绝泛泛而谈，每个段落应包含"事实 → 推演 → 结论"的完整链条
- **假设透明**：对关键假设使用加粗或高亮框提示

**交付物**：Markdown格式研报正文

---

### 步骤6：输出HTML（HTML Rendering）

#### 6.1 渲染方式
- 以 `templates/report-template.html` 为模板，将Markdown研报内容转换为HTML。
- 保持响应式布局，确保在PC与移动端均可正常阅读。

#### 6.2 文件命名
```
reports/主题_YYYYMMDD.html
或
reports/主题_季度.html
```
示例：`reports/新能源汽车产业链2026Q1基本面追踪.html`

#### 6.3 图表处理
- 静态图表：导出为PNG/SVG，存放于 `reports/assets/`，HTML中使用相对路径引用。
- 动态图表：优先使用 ECharts 或 Plotly 嵌入，脚本内联于HTML中。
- 占位图表：在图表区使用 `.chart-placeholder` 样式占位框，并标注"[图表待插入：描述]"

**交付物**：单文件HTML研报

---

### 步骤7：入库与归档（Indexing & Archiving）

#### 7.1 更新 `index.json`
在 `index.json` 中追加研报元数据：

```json
{
  "id": "RPT-YYYY-NNN",
  "title": "研报标题",
  "author": "投研AI中枢",
  "date": "YYYY-MM-DD",
  "tags": ["标签1", "标签2"],
  "category": "宏观/微观",
  "filePath": "reports/文件名.html"
}
```

#### 7.2 更新 `index.html`
确保 Landing Page 的研报列表能自动读取 `index.json` 并渲染新条目。

#### 7.3 Git提交
- 提交信息格式：`docs(report): 新增《研报标题》`
- 推送分支：`origin main`

**交付物**：更新后的 `index.json`、`index.html`、Git提交记录

---

## 三、质量检查清单（Checklist）

- [ ] 选题是否具备可操作的投资结论？
- [ ] 所有核心数据是否标注来源和日期？
- [ ] 宏观 → 行业 → 公司的逻辑链条是否自洽？
- [ ] 是否进行了至少两种数据源的交叉验证？
- [ ] 风险提示是否覆盖了政策、市场、数据、模型四类？
- [ ] HTML是否使用了统一模板，排版是否专业？
- [ ] `index.json` 和 `index.html` 是否已更新？
- [ ] 是否已通过Git提交并推送？
